# 2.5D Map Repair — Comisaría Sala Espera (Vertical Slice)

**Fecha:** 2026-06-08  
**Contexto:** Auditoria `AUDITORIA_JUEGO_ACTUAL_2026-06-08.md` confirmó que los mapas solo tienen `Floor/Walls/Objects` y el motor no puede construir profundidad 2.5D sin las capas `props_back`/`props_front`. Este spec diseña la reparación para el mapa piloto.

---

## Objetivo

Reemplazar `comisaria-sala-espera.json` con un mapa Tiled correctamente autoriado que:

1. Tenga suelo pintado con tiles reales (fin del suelo negro procedural).
2. Tenga capa `props_back` con mobiliario de sala de espera — siempre detrás del jugador.
3. Tenga capa `props_front` con mostrador de recepción — siempre delante, tapa al jugador cuando cruza hacia el norte.
4. Corrija el bug de ordenación en `renderRoom()` que haría que el rectángulo procedural tapara los floor tiles.
5. Sirva de plantilla reutilizable para los otros 13 mapas.

---

## Alcance

**En alcance:**
- `frontend/src/assets/game/maps/comisaria-sala-espera.json` (único mapa esta iteración)

- Fix en `game-world.component.ts` → `renderRoom()`: mover rect procedural a antes de `buildTiledLayers`

**Fuera de alcance:**

## Arquitectura

### Flujo de carga del mapa

```
SimulationPlay
  └─ GameWorldComponent (Phaser Scene: DataDrivenWorldScene)
       └─ setScenarioConfig()  ← ScenarioConfig con comisaria rooms
            └─ renderRoom(roomConfig)          ← path real para comisaría
                 ├─ buildTiledLayers()         ← lee las capas 2.5D
```

`renderComisariaEnvironment()` solo existe en `renderWorld()` (path single-world), que devuelve temprano cuando hay `ScenarioConfig`. Por eso **no hay conflicto** con el render procedural en el path de comisaría.

### Constantes de tiles (TILE_IDS en el script)

Las constantes se agrupan en un dict al inicio del script. Los valores se validan visualmente ejecutando el juego tras generar el mapa; si un tile se ve incorrecto, solo hay que cambiar la constante y re-ejecutar el script.

```python
# tiny-dungeon (firstgid=1, 12 cols)
# GID = 1 + (row-1)*12 + (col-1)
TD = {
    'WALL':       1,    # (row=1, col=1) — stone wall, actualmente en uso
    'FLOOR':      14,   # (row=2, col=2) — stone floor claro — VALIDAR visualmente
    'FLOOR_ALT':  15,   # (row=2, col=3) — variante para chequerboard — VALIDAR
}

# rpg-urban (firstgid=133, 27 cols)
# GID = 133 + (row-1)*27 + (col-1)
RU = {
    'FLOOR_GRAY':    136,  # (row=1, col=4)  — suelo gris interior — VALIDAR
    'FLOOR_GRAY2':   137,  # (row=1, col=5)  — variante suelo       — VALIDAR
    'CABINET_TOP':   214,  # (row=4, col=1)  — parte superior fichero — VALIDAR
    'CABINET_MID':   241,  # (row=5, col=1)  — parte media fichero    — VALIDAR
    'CHAIR':         220,  # (row=4, col=7)  — silla de espera        — VALIDAR
    'TABLE':         222,  # (row=4, col=9)  — mesa pequeña           — VALIDAR
    'COUNTER_TOP':   271,  # (row=6, col=4)  — mostrador (superficie) — VALIDAR
    'COUNTER_FRONT': 298,  # (row=7, col=4)  — mostrador (frente)     — VALIDAR
}
```

> **Convención:** Todos los GIDs marcados `VALIDAR` se comprueban visualmente en el smoke test. Si alguno produce el tile incorrecto, se ajusta la constante y se re-ejecuta. El script es idempotente.

### Definición de capas

#### 1. `floor` (tilelayer, depth 0)

Cubre todo el interior (rows 1–31, cols 1–58). Patrón de chequerboard:
- `(row + col) % 2 == 0` → `RU['FLOOR_GRAY']`
- `(row + col) % 2 == 1` → `RU['FLOOR_GRAY2']`

Coordenadas fuera del interior (borde y exterior) → GID 0 (vacío).

#### 2. `walls` (tilelayer, depth 3)

Borde exterior de 1 tile de grosor + hueco de puerta:

```
Fila 0,  todas las cols:      TD['WALL']   ← pared norte (fondo de sala)
Fila 32, todas las cols:      TD['WALL']   ← pared sur (entrada)
Col 0,   filas 1–31:          TD['WALL']   ← pared oeste
Col 59,  filas 1–31:          TD['WALL']   ← pared este
Excepción (hueco de puerta):
  Col 59, filas 14–16:        GID 0        ← exit-to-consultorio preservado
```

#### 3. `props_back` (tilelayer, depth 2)

Mobiliario en la mitad norte de la sala. Siempre debajo del jugador (depth 2 < actors 1000+Y).

```
Ficheros NW:   filas 1–3, cols 1–4    → RU['CABINET_TOP'] fila 1 / RU['CABINET_MID'] filas 2-3
Ficheros NE:   filas 1–3, cols 55–58  → mismo patrón
Sillas W:      fila 5, cols 2,5,8,11  → RU['CHAIR']  (cada 3 cols)
Sillas W:      fila 7, cols 2,5,8,11  → RU['CHAIR']
Sillas E:      fila 5, cols 47,50,53,56 → RU['CHAIR']
Sillas E:      fila 7, cols 47,50,53,56 → RU['CHAIR']
Mesa central:  filas 4–5, cols 24–28  → RU['TABLE']
Mesa central:  filas 4–5, cols 31–35  → RU['TABLE']
```

#### 4. `props_front` (tilelayer, depth 100 000)

Mostrador de recepción. Siempre sobre el jugador (depth 100k > actores ~2500 máx).

```
Superficie (fila 20, cols 8–36):  RU['COUNTER_TOP']
Frente     (fila 21, cols 8–36):  RU['COUNTER_FRONT']
Brazo L    (filas 19–20, cols 8–9): RU['COUNTER_FRONT']

Paso lateral (cols 37–58): vacío — ruta de acceso a NPCs y puerta este.
```

El jugador (spawn en fila ~28) camina hacia el norte:
- Al cruzar fila 22 (sur del mostrador) → profundidad del jugador ≈ 1022 < 100 000 → mostrador en frente ✅
- El hueco al este (cols 37–58) permite alcanzar los NPCs y la salida sin bloqueo.

#### 5. `lighting` (tilelayer, depth 200 000) — vacía

Todos los GIDs = 0. Establece el contrato para futuras iteraciones.

#### 6. `overlay` (tilelayer, depth 300 000) — vacía

Todos los GIDs = 0. Idem.

#### 7. `Objects` (objectgroup) — preservado exactamente

Los objetos del JSON actual se copian sin modificación:

## Fix en `renderRoom()` — ordenación del rect procedural

### Problema

En el método `renderRoom()` de `game-world.component.ts`, el rectángulo oscuro procedural (depth 0) y la grilla se añaden **después** de `buildTiledLayers`. En Phaser 3, objetos al mismo depth se dibujan en orden de inserción: el rect oscuro (añadido último) aparecería **encima** de los floor tiles (añadidos antes en `buildTiledLayers`), ocultándolos.

```typescript
// Orden actual (INCORRECTO para mapas con floor tiles):
buildTiledLayers(tilemap, tilesets);   // floor layer depth=0, añadido primero
...
this.add.rectangle(...).setDepth(0);   // rect oscuro depth=0, añadido DESPUÉS → tapa floor
this.add.graphics().setDepth(1);       // grilla
```

En `renderWorld()` el orden es correcto (rect oscuro antes de `buildTiledLayers`); solo `renderRoom()` tiene el bug.

### Fix

Mover las líneas del rect oscuro y la grilla al bloque **anterior** al `try { buildTiledLayers... }`:

```typescript
// Nuevo orden (CORRECTO):

// 1. Base procedural (depth 0 y 1) — se añaden PRIMERO
this.add.rectangle(actualMapW / 2, actualMapH / 2, actualMapW - 40, actualMapH - 42, 0x131c28, 1).setDepth(0);
const gfx = this.add.graphics().setDepth(1);
gfx.lineStyle(1, 0x1c2d3e, 0.55);
for (let gx = 16; gx < actualMapW; gx += 32) gfx.lineBetween(gx, 0, gx, actualMapH);
for (let gy = 16; gy < actualMapH; gy += 32) gfx.lineBetween(0, gy, actualMapW, gy);

// 2. Tiles Tiled (floor depth=0 añadido después → encima del rect oscuro)
if (this.assetsLoaded) {
  try {
    const tilemap = this.make.tilemap({ key: roomConfig.tiledMapKey });
    ...
    this.buildTiledLayers(tilemap, tilesets);
    ...
  } catch { ... }
}
```

La grilla (depth 1) seguirá dibujándose sutilmente sobre los floor tiles — aceptable en V1. Se puede suprimir cuando el mapa tenga capas 2.5D en una iteración futura.

## Notas de implementación

**Ajuste de GIDs:** Los GIDs marcados `VALIDAR` en `TILE_IDS` se comprueban al primer smoke. Si el tile es incorrecto, se re-ejecuta el script con el GID corregido. El flujo es: `python generate... → ng serve → smoke → ajustar → repetir`.

**Idempotencia del script:** Cada ejecución sobreescribe el JSON. No requiere cleanup manual.

**Extensión a otros mapas:** Una vez validado el script para comisaria-sala-espera, crear `generate_<mapa>.py` por cada escenario reutilizando las constantes de tileset y la lógica de serialización. El fix en `renderRoom()` beneficia a todos inmediatamente.

**Colisión del mostrador:** El jugador puede cruzar a través del mostrador en V1. La ilusión de profundidad (tapa al jugador) existe sin colisión. Colisión = tarea futura: añadir capa `collision` con tiles coincidentes o usar el Objects layer con rectángulos de colisión.
